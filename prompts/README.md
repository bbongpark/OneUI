# prompts/ — 페르소나 프롬프트

Claude CLI 워커에 주입되는 프롬프트 파일들. 대시보드의 페르소나 편집기가 이 파일들을 읽고 쓴다.

**리뷰 회의의 정체**: SW 담당 부사장급 임원 + 각 부문(경험기획·UX·개발·CXI·PL) 임원이 "이 Feature를 이번 버전에 지원할 것인가"를 결정하는 회의 (결정 = 지원/보류/미지원). 파이프라인은 그 회의에 무엇을 어떤 비중으로 올릴지 **등급**만 정한다 — 등급을 매기는 것은 종합 페르소나 하나뿐이고, 부문 페르소나 4종은 검토 의견만 낸다.

## 구성

| 파일 | 역할 | 실행 시점 |
|---|---|---|
| `_common.md` | 공통 규칙 (비대화형, JSON 출력, needs_human, 등급 체계·역할 분담) — 모든 호출 앞에 자동 주입 | 항상 |
| `persona-experience-planning.md` | 경험기획: 사용자 가치·전략·차별화·시나리오 완결성 (등급 없이 **의견만**) | 리뷰 단계 |
| `persona-ux.md` | UX: 사용성·디자인 일관성·접근성·폼팩터·습관 파괴 | 리뷰 단계 |
| `persona-dev.md` | 개발: 실현성·성능·안정성·보안·의존성·일정 현실성 | 리뷰 단계 |
| `persona-cxi.md` | CXI: VOC 정합성·불만 해소·커뮤니티 반응·지원 비용 | 리뷰 단계 |
| `persona-synthesis.md` | 종합: **등급을 매기는 유일한 페르소나** — 4개 의견 → P0/P1/P2 1개, 인식 차 감지 | 리뷰 단계 (부문 4종 완료 후) |
| `persona-pl.md` | PL: 발표자료 완성도 평가 (문서 완결성 + 발표 준비). 일정 리스크는 코드가 판정 | PL 검사 단계 (PPT 매핑 확정 후) |
| `persona-sw-director.md` | SW담당 임원: 배정된 안건의 예상 결정(지원/보류/미지원) + 확신도 + 예상 질문. **판단 성향 섹션은 회사에서 실제 회의록 기반으로 교체** | 일정 배정 후 (④-2 예상 판정) |
| `aux-title.md` | 제목 생성 — 취합 엑셀에 Feature 이름 열이 없어 변경점을 요약해 제목을 만든다 | 엑셀 인입 직후 (자동) |

## 호출 규약

- 실행 프롬프트 = `_common.md` + 해당 페르소나 파일 + 배치 데이터(JSON)
- 개별 4종은 병렬 개념이지만 워커 큐에서 순차 실행됨 (배치 10~20건 단위)
- 종합은 4종 결과가 모두 모인 Feature만 처리
- 모든 출력은 JSON 파싱 후 `data/<버전>/`에 저장됨 — 스키마를 바꾸면 파서(서버 쪽)도 함께 바꿔야 한다

## 수정 시 주의

- 출력 스키마의 필드명은 대시보드·파서와 계약이다. 문구(관점, 가이드)는 자유롭게 다듬되 스키마 변경은 신중하게.
- 회사 환경에서 실제 데이터로 돌려보며 판정 기준(P0 남발/부족)을 튜닝하는 것이 1차 과제.
