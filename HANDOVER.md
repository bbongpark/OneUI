# 회사 환경 인계 가이드

> 개인 PC에서 뼈대 제작 완료 → 회사 PC에서 실데이터를 붙여 완성하는 단계의 안내서.
> **AI 사용량을 아끼는 것이 목표** — 아래 순서대로 진행하면 대부분 설정·업로드로 끝나고, AI에게 시킬 일은 6·7·9번뿐이다.
>
> **관련 문서**
> - 화면 사용법(운영자·사용자) → [MANUAL.md](MANUAL.md)
> - 코드 구조 철저 설명(코드를 이어받아 개발) → [CODEMAP.md](CODEMAP.md) ⭐ **회사 Claude는 이 파일부터**
> - 설계 배경·의사결정 → [DESIGN.md](DESIGN.md)
> - 짧은 규칙 요약 → [CLAUDE.md](CLAUDE.md)

## 0. 첫 실행 (5분)

```powershell
git clone https://github.com/bbongpark/OneUI "C:\One UI Agent"
cd "C:\One UI Agent"
python scripts\gen_demo_data.py      # 데모 데이터 생성 → 동작 확인용
python server.py                     # → http://localhost:8765
```

- **pip install 불필요** (Python 표준 라이브러리만 사용). Node/npm도 불필요.
- 다른 사람은 브라우저로 `http://<서버PC의 IP>:8765` 접속. 방화벽에서 8765 인바운드 허용 필요.
- 첫 로그인: `관리자` (config/users.json에 등록된 이름만 입장 가능).
- 먼저 **데모 데이터로 전 화면이 뜨는지 확인**한 뒤, 실데이터 인입 전에 삭제:
  ```powershell
  Remove-Item -Recurse -Force data\8.5, data\8.0    # 데모 버전 삭제
  ```
  ⚠️ `gen_demo_data.py`는 실데이터 투입 후 절대 실행하지 말 것 (데모로 덮어씀).
- **`data/`는 git으로 관리되지 않는다** (기밀 + pull 충돌 방지). 백업은 9번 항목의 스크립트로.

## 1. Gemini CLI 연결 ⭐ 최우선

현재 기본 엔진은 `mock`(가짜 결과 생성)이다. 실제 AI를 붙이는 것이 1순위.

1. **설정** 화면 → AI 엔진 → 기본 엔진을 `gemini`로 변경 → 저장
2. **현황판** → `엔진 자가진단` 실행 → **로그·산출물** 화면에서 결과 확인
   - 검사 항목: 비대화형 실행/종료, 순수 JSON 출력, 이미지 파일 읽기, 폴더 파일 읽기
3. 실패 시 `config/engines.json`의 `gemini.command` 플래그를 실제 Gemini CLI 규약에 맞게 수정
   ```json
   "gemini": {
     "type": "spawn",
     "command": ["gemini", "-p", "{prompt_file}", "--output-format", "json"],
     "timeout_sec": 300
   }
   ```
   - `{prompt_file}`은 서버가 임시 파일 경로로 치환한다.
   - 이미지 첨부(PL 검사용)는 명령 뒤에 파일 경로들이 추가로 붙는다 → 이 규약이 다르면 `server/engines.py`의 `_spawn()` 수정.
4. 자가진단 전 항목 통과를 확인한 뒤 다음 단계로.

> **왜 중요한가**: 이 단계가 끝나야 나머지 모든 검증이 진짜 결과로 이뤄진다. mock 상태로 골든셋을 돌리면 무작위 값이 나와 일치율 0%가 정상이다.

## 2. 취합 엑셀 인입 + 스키마 매핑

1. **현황판** → `① 인입 (엑셀·PPT 업로드)` → 버전명(예: `8.5`)과 **DVR 날짜** 입력 → 취합 엑셀 드래그
   - DVR = 개발 일정 리스크 판정 기준일. 나중에 회의 화면에서 수정 가능.
2. **관리 열 선택 창이 자동으로 뜬다** (새 열이 감지되거나 관리 열이 아직 없을 때)
   - 엑셀의 70개 열이 모두 나열되고, 새 열은 **NEW** 뱃지 표시
   - **AI가 참고하고 화면에 표시할 열만** 체크 (검색·전체/해제·새 열만 선택 버튼 제공)
   - 나머지 열의 값도 저장은 되지만 AI에 전달되지 않는다 → 판정 품질·토큰 절감
3. **설정** → 엑셀 스키마 매핑 — 3단계로 진행한다 (선택한 관리 열만 표에 나온다):

   **① 항목 연결** — 시스템이 알아야 하는 항목이 엑셀의 어느 열인지 고른다.
   각 항목 옆에 그 열의 **예시 값**이 표시되므로 맞게 골랐는지 바로 확인할 수 있다.
   | 시스템이 필요한 항목 | 어디에 쓰이나 |
   |---|---|
   | 인덱스 | Feature 식별 · PPT 슬라이드 매핑 기준 |
   | 기능명 | 회의 일정 배정 · 기능별 현황 기준 |
   | AI카테고리 | 분류·필터 기준 |
   | 변경점 | 리뷰의 핵심 입력 · **제목(AI 요약)의 원본** · 자료 보완 규칙 대상 |
   | *UX일정* (선택) | 일정 리스크 판정 — 이게 있어야 개발 일정이 나온다 |
   | *개발일정* (선택) | **일정 리스크 판정 — DVR을 넘으면 리스크** |

   > **Feature 이름 열은 없다** — 목록에 보이는 제목은 AI가 변경점을 요약해 만든다(인입 직후 자동 실행).
   > 제목이 어색하면 `prompts/aux-title.md`를 다듬으면 된다.

   **② 열 용도 지정** — 관리 열마다 체크
   - `필수 기입`: PL 검사가 "이 열이 비면 발표 준비 미완"으로 판정
   - `재리뷰 트리거`: 갱신본에서 이 열이 바뀐 Feature만 재리뷰 (**AI 비용 절감 핵심** — 변경점처럼 판정에 영향을 주는 열만)

   **③ 개발 완료 판정 규칙** (아래 4번 참조)

   - 시트 이름·헤더 행 번호는 카드 상단에서 지정
   - 관리 열 목록을 다시 바꾸려면 `🗂 관리 열 다시 고르기` (언제든 변경 가능)
4. **개발 완료 판정 규칙** (스키마 매핑 카드의 ③단계) — 현황판 "개발 완료 진행률"의 기준
   - **`열에 값이 있으면 완료`** (기본): 판정 열을 `CL`로 지정하면 **CL 번호가 채워진 건**을 완료로 집계.
     `-`, `TBD`, `미정` 등 자리채움 값은 빈 값으로 처리된다. 열 이름이 바뀌면 드롭다운에서 다시 고르면 된다.
     - `값 형식`(선택): 정규식으로 검증. 예: CL이 7자리 숫자면 `^\d{7}$` — 오타나 메모가 섞인 값을 걸러낸다.
   - **`열의 값이 특정 값이면 완료`**: 상태 열 기반으로 판정할 때. 그 열의 실제 값이 건수와 함께 나열된다.
   - **미리보기**가 저장 전에 "이 규칙으로 28/57건(49%)이 완료로 집계됩니다 + 예시"를 보여준다 → 규칙이 맞는지 확인 후 저장.
   - 지정 전에는 현황판 KPI가 "설정 필요"로 표시된다 (틀린 0%를 보여주지 않기 위해).
5. **일정 리스크 확인** — UX/개발 일정 열을 연결하고 DVR을 지정하면 **AI 실행 없이 즉시** 판정된다 (있음/없음):
   - `개발 일정 > DVR` → 있음 ("DVR을 14일 초과") · `UX 일정 미정` → 있음 (개발 일정 산출 불가)
   - `개발 일정 미정` → 있음 (DVR 준수 확인 불가) · `개발 일정 ≤ DVR` → 없음
   - DVR을 바꾸면 전 화면이 즉시 재계산된다 (회의 → 과제 일정). 리뷰 보드의 리스크 뱃지에 마우스를 올리면 근거가 보인다.
6. 저장 후 **리뷰 보드**에서 Feature 목록·상세(관리 열 값)가 제대로 나오는지 확인.

**갱신본이 올 때**: 같은 버전으로 다시 업로드하면 트리거 열이 바뀐 행만 재리뷰 대상이 되고, 나머지는 기존 판정·오버라이드가 그대로 유지된다. 갱신본에서 사라진 행은 삭제되지 않고 "갱신본에 없음"으로 표시된다.

## 3. 발표 PPT 인입 + 매핑 확인

1. **현황판** → `① 인입` → PPT 드래그 (여러 개 동시 가능)
   - 슬라이드 제목의 인덱스(예: `F012`)로 자동 매핑
   - **PNG 렌더링이 자동 실행됨** (PowerPoint COM). 서버 PC에 Office가 있으면 업로드만으로 이미지까지 생성된다.
2. **리뷰 보드** → `PPT 매핑 확인` (미확정 뱃지 표시됨)
   - 미매핑 슬라이드를 Feature에 배정, 잘못 붙은 슬라이드는 재배정/제거
   - **`매핑 확정`을 눌러야 PL 검사를 실행할 수 있다** (안전장치)
3. 슬라이드 인덱스 규칙이 `F###` 형식과 다르면 `server/ingest.py`의 `ingest_ppt()` 매칭 로직 확인.

> PPT 인덱스 규칙이 실제와 다를 가능성이 가장 큰 리스크 지점. 첫 PPT 업로드 시 매핑률을 반드시 확인할 것.

## 4. 페르소나 튜닝 (실데이터 확인 후)

1. **페르소나** 화면에서 프롬프트 7종 + 보조 6종 편집 가능. 저장 시 자동 백업.
2. **`prompts/persona-sw-director.md`의 "판단 성향" 섹션을 실제 회의록 기반으로 교체** — 현재는 일반적인 빅테크 임원상으로 채워둔 초안이다. 지난 회의록에서:
   - 실제로 자주 나온 질문
   - 반려/보류 사유의 패턴
   - 중시하는 지표
   를 뽑아 6개 성향 항목을 다시 쓴다.
3. **등급 규칙 채우기** — `config/grade_rules.json` ⭐ 실데이터를 보고 확정할 항목
   - `doc_fix`: 변경점이 부실하면 **자료 보완 필요(DOC)**. AI 앞단에서 걸러 호출하지 않는다.
     **8번의 변경점 템플릿을 확정한 뒤 `require_all`을 그 항목과 일치**시킬 것 (기본값: "변경 전"·"변경 후" 포함 + 20자 이상)
   - `share.rules`: **AI가 P2(서면보고)로 판정한 건 중** 단순 공유로 내릴 규칙. 예시 규칙이 하나 들어 있으니 실제 열·값으로 교체 (op: equals/contains/in/empty/not_empty/regex)
4. 리뷰를 돌려보고 P0/P1/P2 분포가 이상하면 `persona-synthesis.md`의 **판정 규칙**을 조정 (등급을 매기는 건 종합 하나뿐).
   - 등급은 **① 파급력 + ② 대면 필요성(결정·쟁점 또는 공유·정렬)** 두 축으로 정한다. 공유만 필요해도 P1. 공유 사유로 P1이 과하게 늘면 판정 규칙 3번의 문턱을 조인다.
   - 부문 페르소나(경험기획·UX·개발·CXI)는 **등급 없이 검토 의견만** 낸다 — 의견에 파급력·타 부문 공유 필요를 담게 되어 있다(각 파일의 "의견 가이드").
   - 진행/중단(go/no-go)은 리뷰가 판단하지 않는다 — 회의에서 임원이 결정하고 회의록에서 추출된다
   - 발표자료 완성도는 **③ PL 검사**가 따로 평가한다 (PPT 필요)
   - **SW담당 예상 판정**은 PL이 '준비됨'으로 판정한 안건에 대해서만 돈다 (자료 없으면 예측 근거 없음). 예측·근거는 리뷰 보드 상세 모달 맨 아래에 표시된다.
4. ⚠️ **출력 스키마(JSON 필드명)는 건드리지 말 것** — 서버 파서와의 계약이다. 관점·기준·어조만 수정.

## 5. 골든셋 등록 (품질 측정의 기반)

1. **페르소나** 화면 하단 → `골든셋 엑셀 양식 받기`
2. **실제 취합 엑셀에서 행 20~30개를 복사** → 마지막에 3개 열 추가:
   - `정답등급`: P0 / P1 / P2
   - `정답결정`: 지원 / 보류 / 미지원 (회의 결정 3종)
   - `메모`: 판정 이유 (선택)
   > 열 이름을 바꾸고 싶으면 `config/excel_schema.json`에 `golden_columns` 추가.
3. 드롭존에 업로드 → `골든셋 실행` → 등급/권고 일치율 확인
4. 페르소나를 수정할 때마다 재실행 → **일치율이 오르는지로 판단**. 감이 아니라 측정.
5. 엔진 비교(Gemini vs Claude)에도 같은 방법 사용.

## 6. PLM API 연동 (개발 필요)

현재는 mock이다. 교체 지점:

| 위치 | 현재 | 교체 후 |
|---|---|---|
| `server/api.py` → `api_plm_advance()` | 상태를 pending→sent→in_progress→done으로 수동 진행 | 실제 PLM API 호출 (액션 아이템 전송 / 상태 폴링) |

- 액션 아이템 구조: `{id, feature_index, action, owner_dept, due, plm_status, plm_id, report_needed, followup_scheduled}`
- `plm_status`는 `pending|sent|in_progress|done` 4단계 유지 (화면·판단 로직이 이 값에 의존)
- 인터페이스만 지키면 나머지(보고 필요 판단 → 후속 보고 배정 → 회의 재진입)는 그대로 동작한다.
- 폴링이 필요하면 `server/jobs.py`에 주기 작업을 추가하거나 Windows 작업 스케줄러에서 API를 호출.

## 7. 보고 PPT 템플릿

1. 회사 표준 보고 양식 pptx를 `templates/`에 넣는다.
2. 텍스트 상자·표 안에 자리표시자를 심는다: `{{버전}}`, `{{전체건수}}`, `{{P0건수}}`, `{{P1건수}}`, `{{P2건수}}`, `{{리뷰진행률}}`, `{{PL통과율}}`, `{{결정완료율}}`, `{{리스크건수}}`, `{{총평}}`, `{{생성일}}`
3. `server/jobs.py`의 `job_report_ppt()`를 `scripts/ppt_fill.ps1` 호출로 전환 (현재는 마크다운 요약만 생성).
4. 자리표시자를 추가하려면 `job_report_ppt()`에서 값을 채워 `report_fill.json`으로 넘기면 된다.
5. 템플릿 파일이 늘어나면 산출물 버튼도 늘어난다.

## 8. 변경점 템플릿 확정

- `templates/변경점_템플릿.md`은 초안이다. **실제 변경점 열의 내용을 보고** 다듬을 것.

| **자동 시작** | 서버 PC 재부팅 시 `python server.py` 자동 실행 (작업 스케줄러 "시작 시" 트리거) |
| **인증/보안** | 현재는 이름 입력만. 사내 LAN에서 IP를 아는 누구나 접속 가능하다. 미출시 Feature 정보의 기밀 등급을 확인하고, 필요하면 `api_login()`을 사내 인증으로 교체. **보안팀 확인 필요 항목.** |
| **사용자 등록** | 설정 → 사용자·역할에서 20명 등록. 관리자/일반 구분. |
| **알림 확장(선택)** | `store.notify()`가 발행 단일 지점 — 사내 메신저 연동을 여기에 붙이면 된다. |
| **사용량 모니터링** | `data/usage.json`에 호출별 토큰·비용 기록됨. Claude를 병행하면 이 기록으로 예산 정책을 되살릴 수 있다 (제거된 상태). |

## 알아둘 제약

1. **PPT 영상 재생 미지원** — 슬라이드는 PNG 정지 이미지로만 보인다. pptx의 `ppt/media/`에서 mp4를 추출해 `<video>`로 재생하는 확장은 가능하다 (미구현).
2. **PNG 렌더링은 Office 필수** — 서버 PC에 PowerPoint가 없으면 텍스트 매핑만 되고 이미지 뷰어·비전 검사가 동작하지 않는다.
3. **개인 PC에서 미검증 경로**: PowerPoint COM 렌더링(Office 없음), Gemini CLI 실제 호출. 회사에서 첫 실행 시 확인 필요.
4. **동시 실행은 1개** — 작업 큐가 직렬이라 500건 전체 리뷰는 수 시간 걸린다. 야간 실행 권장.
5. **질의 성능** — 데이터가 수년치로 커지면 키워드 검색이 느려질 수 있다. 그때 인덱스 도입 검토.

## 문제 해결

| 증상 | 확인 |
|---|---|
| 화면이 안 뜸 | `python server.py` 콘솔 오류, 8765 포트 충돌 |
| 다른 PC에서 접속 불가 | 방화벽 8765 인바운드, 서버 PC IP 확인 |
| AI 결과가 이상함 | 설정에서 엔진이 `mock`이 아닌지 확인 → 자가진단 실행 |
| 리뷰가 재실행 안 됨 | 캐시 3중 규칙 — 입력 해시/트리거 열/프롬프트 해시가 모두 같으면 건너뛴다 |
| PL 검사 실행 거부 | 리뷰 보드에서 PPT 매핑 확정 필요 |
| 판정이 사라짐 | `data/<버전>/reviews.json` 직접 열람 (모든 상태가 파일) |
| 동시 편집 충돌 | 낙관적 잠금 — 새로고침 후 재시도 |
