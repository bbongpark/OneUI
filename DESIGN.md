# One UI Agent — 설계 문서

> 이 문서는 프로젝트의 단일 기준점(source of truth)이다. 구현·확장·인수인계 시 항상 이 문서부터 읽는다.
> 작성: 2026-07-15. 개인 PC에서 뼈대 설계 완료, 회사 환경 이관 전 상태.

## 1. 목적

One UI Feature의 **버전 탑재 여부를 결정하는 부사장급 리뷰 회의**를 준비·운영·후속 관리하는 파이프라인을 AI CLI 페르소나로 자동화하고, 약 20명이 함께 쓰는 웹 대시보드로 조종·관제한다.

리뷰 회의의 정체: SW 담당 부사장급 임원 + 각 부문(경험기획·UX·개발·CXI·PL) 임원이 "이 Feature를 해당 버전에 진행할 것인가(go/no-go)"를 결정하는 의사결정 회의. 페르소나들의 산출물은 이 결정을 준비하는 부문별 권고안 초안이다.

## 2. 개발 전략과 환경 제약

| 항목 | 내용 |
|---|---|
| 개발 이원화 | 개인 PC에서 뼈대 완성 → git 저장소 경유로 회사 이동 → 회사에서 살 붙임 (회사 Claude는 사용량 제한이 있어 최소로 사용) |
| 회사 PC 환경 | Node.js 없음, npm 제한적/불확실, **Python 있음, MS Office 있음**, Gemini CLI 사용 가능(토큰 제한 거의 없음), Claude Code(월 $300 상당) |
| 스택 원칙 | **설치 제로**: Python 표준 라이브러리만(서버), 바닐라 HTML/JS(빌드 없음, 차트 라이브러리는 단일 파일 vendor 커밋), PowerShell COM(엑셀·PPT), 상태는 전부 JSON 파일 |
| 동시 사용 | 약 20명, 사내 LAN. 서버 PC 1대에서 `python server.py` 실행, 나머지는 브라우저 접속. 사용자는 AI 계정 불필요 |
| 보안 | 뼈대 범위 밖 (이름/사번 입력 식별만). 실제 인증은 어댑터로 분리해 회사에서 결정 — **회사 확인 필요 항목** |

## 3. 파이프라인 (버전 단위 순환)

```
[버전 생성] + 과제 마일스톤 입력 ({이름, 날짜} 자유 목록, 관리자 편집)
    │
 ① 인입      엑셀 취합본 1개 + PPT 여러 개 + 참고자료(벤치마킹/동향 문서)
    │         · 현황판 "① 인입" 버튼 → 드래그&드롭 업로드 (새 버전명 입력 시 버전 생성)
    │           엑셀 파싱은 서버 내장(표준 라이브러리) — 스키마 매핑·증분·재등록 감지
    │         · PPT 텍스트 매핑은 업로드 즉시 · PNG 렌더링은 업로드 시 자동으로 큐 등록
    │           (PowerPoint COM — Office 없는 PC면 우아하게 건너뛰고 알림)
    │         · PPT 인입 시 매핑은 "미확정" 상태 — 리뷰 보드 'PPT 매핑 확인'에서
    │           사람이 검토(재배정/제거/미매핑 배정) 후 확정해야 PL 검사 실행 가능
    │         · 슬라이드 제목의 인덱스 번호 ↔ 엑셀 행 자동 매핑
    │           (인덱스 없음/오류 → AI 비전 추정 + 대시보드 수동 지정)
    │         · 이전 버전 rejected/defer 항목과 매칭 → "재등록" 표시 + 당시 근거 링크
    │         · 갱신본: review_trigger_columns가 바뀐 행만 재리뷰 대상, 나머지 캐시 유지
    ▼
 ② 리뷰      개별 페르소나 4종(경험기획·UX·개발·CXI)
    │           → 등급(P0/P1/P2) + 진행 권고(go/conditional_go/defer/no_go) + 핵심 질문
    │         종합 페르소나 → 최종 등급 + 종합 권고 + 갈린 지점(divergent) [사람 오버라이드]
    │         · P2 = 서면보고: PPT 등록까지만, 별도 승인 흐름 없음, 회의 제외
    ▼
 ③ PL 검사   문서 완결성(필수 열, 변경점 템플릿 준수)
    │         + 발표 준비(우측 상단 표 기입, VOC 정량 표기 — 슬라이드 PNG 비전 검사)
    │         ※ 일정 리스크는 AI가 아니라 코드가 규칙으로 판정 (§ 일정 리스크 규칙)
    ▼
 ④ 일정 배정  달력에서 회의일 지정 (회의는 하루 한 번 = 날짜당 슬롯 1개, 보통 3일)
    │         → AI가 건별 소요시간 추정 [사람 수정]
    │         → 중요도×제안부서 기준 배정 (추정 없으면 기본 1h≈12건)
    │         [보드에서 드래그앤드롭으로 회의일 간 이동·미배정, 용량 초과는 경고만
    │          — 자동 밀어내기 금지]
    ▼
 ④-2 예상 판정  SW담당 임원 에이전트가 배정된 안건 순회:
    │         건별 예상 결정(go/conditional_go/defer/no_go) + 확신도 + 예상 근거
    │         + 예상 질문(발표 부문의 사전 대비 자료)
    │         · 페르소나 판단 성향은 회사에서 실제 회의록 기반으로 교체 (초안 작성됨)
    ▼
 ⑤ 회의      회의록 입력 창(붙여넣기) → AI 추출:
    │           건별 결정(go/no_go/defer/조건) + 액션 아이템 [사람 확인 후 확정]
    │         · no_go → rejected: 모든 KPI 모수에서 제외, 재등록 추적용으로만 보관
    │         · 실제 결정 확정 시 ④-2 예상과 자동 비교 → 적중률 기록
    │           (SW담당 페르소나 품질 지표 — 회의록이 쌓일수록 개선)
    ▼
 ⑥ PLM      액션 아이템 전송 → 처리 결과 수신 (mock 어댑터, 회사에서 실제 API로 교체)
    │         → AI "보고 필요" 판단 [사람 수정]
    │         → 보고 필요 항목: 가장 빠른 가용 슬롯 우선 배정(후속 보고 태그) → ④ 순환
    ▼
 ⑦ 인사이트  전체 리뷰 데이터 + 참고자료 기반 리포트: 버전 주요 지향점, 트렌드 정합성
    │         · 웹 검색 없음. 참고자료 파일 기반, 주장마다 출처 표기 필수
    │         · 자료 없으면 "모델 지식 기준" 명시
 ⑧ 산출물    취합완료/진행보고 PPT 생성 — templates/의 pptx에 {{자리표시자}} 치환
              · 수치·표 = 코드가 결정적으로, 서술(총평 등) = AI 작성 [사람 확인]
```

### Feature 상태 머신

`인입 → 리뷰 중 → 회의 대기 → 결정됨(go | conditional_go | defer | rejected) → 추적(액션·PLM) → 완료`

- **rejected**: 통계·KPI 모수에서 완전 제외. 차기 버전 재등록 감지에만 사용.
- **defer**: 차기 버전 이월 추적 (재등록 감지와 동일 메커니즘).
- 화면들은 상태를 각자 계산하지 말고 이 상태 필드만 읽는다.

### 공통 패턴: AI 제안 → 사람 확정

모든 AI 판단(종합 등급, 권고, 소요시간, 결정 추출, 보고 필요, 리스크)은 사람이 수정 가능하고, 수정 시 원값·수정자·사유·시각이 이력으로 남는다. AI가 판단 불가한 항목은 `status: "needs_human"`으로 반환되어 사람 확인 목록에 오른다. 새 판단 단계를 추가할 때는 이 틀을 복제한다.

## 4. 실행 계층

### 4.1 멀티 엔진 어댑터

```
[작업 큐] → [엔진 어댑터 인터페이스]
               ├─ gemini (기본): spawn 모드 — 호출마다 프로세스 실행
               └─ claude (예비): persistent 모드 — 상주 워커, stdin/stdout JSON 스트림,
                                  N작업마다 자동 재시작(컨텍스트 누적 방지), 사망 시 재기동
```

- **기본 엔진 = Gemini CLI** (토큰 제한 거의 없음, 사용자가 JSON·PNG 테스트 통과 확인). 문제 발생 시 설정 한 줄로 Claude로 전환.
- 페르소나별로 엔진·모델을 설정 파일에서 지정 가능 (`config/engines.json`).
- 어댑터 계약: `(프롬프트, 첨부 파일 목록, 허용 폴더) → JSON 결과 + 사용량`. 파이프라인 본체는 엔진을 모른다.
- **엔진 자가진단**: 대시보드 버튼 → 설정된 엔진에 대해 실측 검증 ①비대화형으로 JSON만 반환 ②이미지 읽기 ③지정 폴더 파일 읽기 ④멈춤 없이 종료. 통과/실패 표시. 회사에서 엔진 붙일 때 말이 아닌 실측으로 확인.

### 4.2 비대화형 원칙

- CLI는 어떤 경우에도 사람에게 질문할 수 없다: 권한 프롬프트 차단(허용 도구·폴더 사전 지정), 프롬프트 공통 규칙("질문 금지, 판단 불가 시 needs_human"), 작업 타임아웃(초과 시 실패 처리 + 알림, 큐는 계속).

### 4.3 작업 큐

- 모든 AI 작업은 큐로 직렬화. 동시 클릭해도 실행은 하나씩. 요청자·시각 기록.
- 질의는 전용 슬롯 분리 (파이프라인 실행 중에도 질의 응답 가능).
- 배치: 리뷰·검사는 Feature 10~20건 묶음 호출, 결과는 JSON 배열 파싱.
- 전체 실행 수 시간 소요 허용 (500건 기준). 병렬화하지 않음.

### 4.4 캐시 3중 규칙 (증분 실행)

1. **입력 해시**: 결과 JSON에 입력(행 데이터) 해시 저장 → 같은 입력은 재실행 안 함.
2. **트리거 열**: 갱신 엑셀에서 `review_trigger_columns`(설정)가 바뀐 행만 재리뷰. 다른 열만 바뀌면 데이터 갱신 + 리뷰 캐시 유지. 사람 오버라이드가 있는 행의 입력이 바뀌면 오버라이드는 유지하되 "입력 변경됨" 플래그로 재확인 목록에 표시.
3. **프롬프트 버전 해시**: 결과에 판정 당시 페르소나 프롬프트 해시 저장 → 페르소나 수정 시 **그 페르소나 결과 + 종합만** 재실행. 나머지 캐시 유지.

### 4.5 사용량

- 모든 호출의 토큰·비용은 data/usage.json에 기록 (엔진 무관 상시).
- 예산 게이지·잠금 정책은 UI에서 제거됨 (2026-07-16, Gemini 기본 운용 확정에 따라) — Claude를 대량 사용하게 되면 usage.json 기록을 근거로 정책을 되살릴 수 있다.

## 5. 데이터 구조

```
One UI Agent/
├─ DESIGN.md                  ← 이 문서
├─ CLAUDE.md                  ← AI 어시스턴트용 안내 (구현 시 작성)
├─ server.py                  ← Python 표준 라이브러리 서버 (진입점)
├─ server/                    ← 서버 모듈 (큐, 엔진 어댑터, 파일 락, 알림 발행 등)
├─ web/                       ← 바닐라 JS 대시보드 (위젯 = 파일 1개 패턴)
│   └─ vendor/                ← 차트 등 단일 파일 라이브러리 (git 커밋)
├─ prompts/                   ← 페르소나 프롬프트 (작성 완료: 7종 + README)
├─ scripts/                   ← PowerShell COM (excel→json, ppt→png, ppt 치환, 문서 텍스트 추출)
├─ templates/                 ← 변경점 템플릿, 보고 PPT 템플릿 ({{자리표시자}} 규약)
├─ config/
│   ├─ excel_schema.json      ← 논리 필드↔실제 열 매핑, required_columns, review_trigger_columns
│   ├─ engines.json           ← 엔진·모델 페르소나별 지정, 예산 정책
│   └─ users.json             ← 사용자·역할(관리자/일반)
├─ golden/                    ← 골든셋 (사람 정답 판정 세트 + 일치율 측정)
└─ data/
    ├─ usage.json             ← 사용량 원장 (버전 무관)
    ├─ notifications.json     ← 알림 피드
    └─ <버전>/                 ← 예: 8.5/
        ├─ features.json      ← 엑셀 추출본 + 상태 머신 + 매핑 + 해시들
        ├─ reviews.json       ← 페르소나별 결과 + 종합 + 오버라이드 이력
        ├─ pl_checks.json     ← PL 검사 결과
        ├─ schedule.json      ← 마일스톤 + 회의 슬롯 배정
        ├─ meetings.json      ← 회의록 원문 + 추출 결과(결정·액션) + 확정 기록
        ├─ prediction_stats.json ← SW담당 예상 vs 실제 적중률 기록
        ├─ actions.json       ← 액션 아이템 + PLM 상태
        ├─ slides/            ← Feature별 슬라이드 PNG
        ├─ references/        ← 참고자료 원본 + 추출 텍스트 캐시
        ├─ output/            ← 생성된 보고 PPT
        └─ insight.md         ← 인사이트 리포트
```

- 모든 저장 레코드에 버전·feature_index·시각을 포함한다 (질의 출처 표기의 재료).
- 서버가 유일한 쓰기 주체. 쓰기 락으로 직렬화. 편집 저장은 낙관적 잠금(읽은 버전 불일치 시 거부 + 새로고침 안내).
- 지난 버전 폴더는 읽기 전용 조회.

### 엑셀 스키마 = 블랙박스

실제 열 정보는 제공 불가(기밀, 약 70열). 코드에 열 이름 하드코딩 금지. `excel_schema.json`이 논리 필드(feature_index, feature_name, department, dev_status, …)와 실제 열을 연결한다. 개발·검증은 그럴듯한 가짜 엑셀로 한다.

**관리 열(managed_columns)**: 70열 중 참고·관리 대상 열만 선택한다.

- **엑셀 업로드 직후 "관리 열 선택" 창이 자동으로 열린다** (새 열 감지 시 또는 관리 열 미설정 시). 전체 열 목록 + 새 열 NEW 뱃지 + 검색/전체/해제/새 열만 선택.
- 같은 창을 설정 화면의 `🗂 관리 열 선택` 버튼으로 언제든 다시 열 수 있다.
- **설정의 스키마 매핑 표에는 관리 열만 표시** — 70행이 아니라 선택한 ~10행만 다루면 된다 (논리 필드 지정 열은 항상 포함).
- 페르소나(리뷰·PL·예상판정·소요추정)에게는 관리 열 + 논리 필드 열만 직렬화해 전달하고, 리뷰 보드 상세에도 관리 열만 표시한다. 미선택 = 전체 사용.

### 일정 리스크 규칙 (AI 아님 — 코드가 결정적으로 판정)

실제 운영 규칙: **UX 일정이 있어야 개발 일정이 나오고, 개발 일정이 DVR을 넘으면 리스크.**

| 조건 | 판정 | 근거 표시 |
|---|---|---|
| 개발 일정 > DVR | **high** | "개발 일정(9-11)이 DVR을 14일 초과" |
| 개발 일정 ≤ DVR | normal | "DVR 이내 (N일 여유)" |
| 개발 일정 없음 + UX 일정 없음 | **high** | "UX 일정 미정 — 개발 일정을 산출할 수 없음" |
| 개발 일정 없음 + UX 일정 있음 | caution | "개발 일정 미정" |
| DVR 미설정 / 열 미지정 | unknown | 설정 안내 |

- **DVR**은 과제(버전) 단위 값 — 인입 시 입력하고 **회의 화면 "과제 일정"에서 수정**(관리자). 수정 즉시 전 화면의 리스크가 재계산된다(AI 재실행 불필요).
- UX 일정·개발 일정 열은 스키마 매핑의 선택 항목(`ux_schedule`, `dev_schedule`).
- 판정은 `App.scheduleRisk(f)` 한 곳. 날짜 파서는 `2026-08-28` / `2026.8.5` / `2026년 9월 3일` 등을 처리하고 **로컬 자정**으로 만든다(ISO를 `new Date()`에 넘기면 UTC로 해석돼 하루 밀림).
- PL 검사를 돌리기 전에도 즉시 판정된다 — 엑셀 데이터만으로 계산되므로.

**열 이름도 값도 판정 규칙도 하드코딩 금지**: 현황판의 개발 완료 진행률은 `dev_done_rule`(설정)로 판정한다.

- `mode: "filled"` — 지정 열에 값이 있으면 완료. **실제 운영 규칙: `CL` 열에 CL 번호가 있으면 개발 완료.** 열 이름은 바뀔 수 있으므로 드롭다운에서 선택하고, `-`/`TBD`/`미정` 등 자리채움 값(`App.PLACEHOLDERS`)은 빈 값으로 취급한다. `pattern`(선택)으로 값 형식을 정규식 검증.
- `mode: "values"` — 지정 열의 값이 목록에 있으면 완료 (상태 열 기반 판정용). 설정 화면이 그 열의 실제 값을 건수와 함께 나열.
- 판정은 `App.isDevDone(feature)` 한 곳에서만 하고, 설정 화면은 **저장 전 미리보기**(N/M건, %, 예시)로 규칙을 검증하게 한다. 미지정 시 KPI는 "설정 필요" — 틀린 0%를 보여주지 않는다.
